The backend is production-grade — the security model, service layer, and database are all well-engineered. The frontend is where the project stalls. Several critical pieces are either missing entirely or architecturally wrong.

Backend — what's solid
The tenant context pipeline is the strongest part of the codebase. Every API route calls requireTenantContext(req) which extracts and verifies the JWT via getUser() (not getSession()), resolves the organization from cookie → profile → membership fallback, and validates the caller is actually a member of that org before any data is touched. The authorize() call on top of that enforces RBAC through a clean permission matrix in permissions.ts. This is exactly the right architecture.
The RLS migration (20260316000100) is comprehensive. All nine tables have RLS enabled with is_org_member() and org_role() as security-definer helper functions. The policies are correctly written — they use the helper functions rather than inlining the subquery each time, which means there's one place to update if the membership logic ever changes. The messages table migration adds a CHECK constraint enforcing that recipient_id and project_id are mutually exclusive, which is a nice schema-level guard.
Service layer is cleanly separated. Every service function scopes queries with organization_id and is("deleted_at", null). Soft-delete cascades from projects to tasks. Assignment conflict detection with assertNoDuplicateActiveAssignment prevents duplicate active assignments. The hydrateAssignmentRows pattern that enriches assignment rows with profile and task data in two parallel queries is efficient.
Zod validation is on every API route input, with centralized schemas in lib/validation/. The mapUnknownError function correctly catches Postgres error codes (like 23505 for conflicts) and maps them to typed HTTP responses via fail() and ok().

Frontend — the real story
LeftSideBar.tsx is GitHub's 404 HTML page. The entire file is raw HTML from GitHub's error page, including a unicorn image as a base64 data URL. This means the app shell has no navigation at all. This is the single most disruptive bug in the codebase — it affects every authenticated page.
Auth forms are static. Both login/page.tsx and signup/page.tsx render forms with <Button type="submit"> that do nothing. There's no Server Action on the form and no client-side onSubmit. Users cannot authenticate at all through the UI. The backend auth flow via Supabase works fine — it's purely a missing frontend wire-up.
Client-side Supabase writes in pages. The handleCreate function in organizations/[orgId]/projects/page.tsx calls supabase.from("projects").insert(...) directly from the client. This bypasses the service layer entirely. The correct createProjectAction Server Action exists and is even imported at the top of the file, but is unused. The pattern is repeated in UserTasksView.tsx which queries tasks with no organization_id filter, relying entirely on RLS rather than scoping the query correctly.
Inbox is entirely placeholder. InboxView.tsx renders two hardcoded strings. ComposeMessagePopup.tsx says "Messaging is not available yet." The API — /api/messages, /api/messages/send — is complete, tested, and secured. The RLS policies exist. It just needs a frontend component that actually calls those endpoints.
Employee detail page loads the wrong org. In the useEffect in [employeeId]/page.tsx, it fetches the org with supabase.from("organizations").select("id").single() — no filter at all. This returns a random org rather than the one from the URL segment [orgId]. Any task created from this page would land in the wrong organization.
TopBar returns null for most routes. It only renders for five hardcoded path patterns checked with pathname.includes(...). Any org-scoped route that doesn't match — like /organizations/[orgId]/analytics — gets no top bar at all. The page renders fine, but the bar is just absent.

Architecture observations
The two-layout system is confusing. There's an (app)/layout.tsx that sets up a DashboardProvider, ToastProvider, LeftSideBar, and TopBar — but then there are separate server-component pages under /projects, /tasks, /dashboard, and /settings that also work and use Server Actions and requireOrgContext. These two surfaces aren't connected. A user logging in via the auth forms would land at /dashboard (which uses the server-component surface), while the employee and project management lives under /organizations/[orgId]/... (which uses the client-component shell). There's no routing logic that ensures users end up in the right surface or that the [orgId] segment is always populated correctly.
removeProjectMember in src/actions/project/removeProjectMember.ts imports deleteAssignment and listAssignments from services/resource/assignment.service but there's a name collision — the file also has local variables with those names. This would cause a runtime error when removing a project member triggers assignment cleanup.
The proof column is referenced in UserTasksView.tsx (status: "done", proof: proof) but doesn't exist in the tasks table schema in database.ts. That supabase.update() call will silently succeed but the column value is discarded.
